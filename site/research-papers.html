STAPL Research Paper Map

Use this page as a guided bridge between the ideas on the teaching pages and the research papers listed on the public STAPL publications page.

Core architecture papers

STAPL: Standard Template Adaptive Parallel Library (2010)

Start here for the broadest overview: unified support for shared and distributed memory, major components, and the role of the runtime system.

The STAPL pView (2010)

Read this when you want the strongest explanation of why views are central to STAPL's programming model and why they generalize iterator-style access.

The STAPL Parallel Container Framework (2011)

Read this to understand how distributed containers are assembled and why extensibility matters for performance and expressivity.

The STAPL Parallel Graph Library (2012)

Read this once the core abstractions make sense and you want to see how they are specialized for graphs at scale.

Performance and runtime papers

KLA (2014)

This paper is about synchronization tradeoffs: reducing expensive global barriers while tolerating controlled redundant work.

STAPL Skeleton Framework (2014) and composed skeletons (2015)

These papers connect STAPL to dataflow-graph reasoning and higher-order composition of parallel patterns.

STAPL-RTS (2015)

This is the runtime paper to read when you want to understand how application semantics can guide lower-level execution choices.

Asynchronous nested parallelism (2015)

This paper matters if you care about irregular applications, divide-and-conquer structure, and avoiding rigid phase boundaries.

Concepts extracted from the Skeleton Framework paper

From the LCPC 2014 chapter in the proceedings PDF, the most teaching-relevant STAPL ideas are these: STAPL hides processing elements, supports asynchronous communication through RMIs, exposes a data flow engine called PARAGRAPH, and translates skeletons into parametric data flow graphs whose compositions become point-to-point dependencies rather than barrier-heavy bulk phases.

RMIs on shared objects

Useful teaching angle: user code stays high-level while communication is expressed through runtime-managed remote method invocations.

PARAGRAPH

The paper presents PARAGRAPH as the data flow engine that executes task-graph style parallelism inside STAPL.

Parametric dependencies

A skeleton is not just an opaque primitive. It is modeled as a graph of input-output relationships that can be expanded once input data is known.

Point-to-point composition

This is the big performance idea: avoid unnecessary global synchronization by wiring skeleton compositions as direct graph dependencies.

Skeleton-to-dataflow graph

This visualization turns the paper's main idea into a simple study picture. Step through the stages to see how a high-level skeleton composition becomes a dataflow graph with explicit dependencies that can overlap communication and computation.

Specification view: think in terms of reusable skeletons like `map`, `zip`, and `reduce`, not raw message passing.

Graph-systems thread

The graph papers form a mini-story inside STAPL research. The 2012 graph-library paper introduces the high-level framework and multiple graph processing paradigms. KLA 2014 studies synchronization tradeoffs. The 2015 communication-reduction paper studies hierarchical coarsening and redundancy reduction. The 2016 bounded-asynchrony paper explores explicit performance-versus-accuracy tradeoffs in graph traversal workloads.

Why this matters for the teaching site

  • The memory-model pages can now explain that STAPL is not only about distributed storage, but also about task-graph execution.
  • The C++ connections page can point out that skeletons are higher-order reusable algorithm forms, closer to generic programming than raw MPI code.
  • The visual model becomes clearer: views explain access structure, while skeleton graphs explain execution structure.
